## 1. 트리아지 원칙

1. **"만들기 전에, 있는 것부터"** — 상용·기성 도구로 해결되면 만들지 않는다. Sandbox는 마지막 선택지다
2. **최소 수단 우선** — 노코드로 되면 노코드, 로우코드로 되면 로우코드. 풀코드(Sandbox)는 앞 단계로 해결 불가할 때만
3. **개인용은 배정하지 않는다** — Sandbox는 조직(팀 이상)의 생산성 향상을 위한 에이전트 전용
4. **T2Y 정렬** — Sandbox에서 만들어지는 에이전트는 T2Y 달성에 기여하는 것이어야 한다
5. **자격은 게이트다** — 요건 미충족자는 거절이 아니라 교육 경로로 안내한다 (재도전 유도)

## 2. 도구별 프로파일

### 2.1 A.Biz — SK 사내 AI 에이전트

| 항목 | 내용 |
|:---|:---|
| 성격 | 사내 공통 업무 AI 에이전트 + 간단한 노코드 챗봇 제작 |
| 할 수 있는 것 | 회의실 예약, 미팅(회의록) 작성 등 사내 공통 업무 기능. n8n류의 **간단한 노코드 RAG 챗봇 제작** — 문서를 올려 질의응답하는 팀 챗봇 수준 |
| 할 수 없는 것 | 복잡한 다단계 로직, 외부 시스템 연동 자동화, 서비스형 배포·운영 |
| 적합 신호 | "우리 팀 문서로 질의응답 챗봇을 만들고 싶다", "사내 공통 업무(회의·예약)를 AI로 처리하고 싶다" |
| 부적합 신호 | 여러 시스템을 연결한 자동 실행, 커스텀 UI·API 서비스 필요 |

### 2.2 MS Copilot — 개인 생산성 (Work IQ 기반)

| 항목 | 내용 |
|:---|:---|
| 성격 | M365(Outlook·SharePoint·OneDrive·Teams) 데이터를 이해하는 **개인 생산성 보조** |
| 할 수 있는 것 | 메일·문서·회의의 조회·요약·검색, 초안 작성, 프롬프트 세팅 기반 반복 생산성 작업(예: 주간 메일 요약) |
| 할 수 없는 것 | **실행(action) 기능 없음** — 메일 자동 발송, 문서 작성 후 SharePoint 자동 저장 같은 "대신 실행"은 불가* |
| 적합 신호 | "내 메일·문서를 요약/정리하고 싶다", 수혜자가 본인 1인 |
| 부적합 신호 | "자동으로 보내/저장해/처리해 줘"(실행 필요) → Copilot Studio 검토, 팀 공용 서비스 필요 |

> \* 2026년 기준 MS는 Work IQ 에이전트로 일부 실행 기능(메일 발송·일정 등록 등)을 제공하기 시작했으나, 테넌트 정책·라이선스에 따라 가용 범위가 다르다. **사내 기준은 "조회·생성 보조, 실행 없음"으로 판정**하고, 사내 테넌트에서 실행 기능이 활성화되면 본 기준을 개정한다.

### 2.3 Copilot Studio — 로우코드/노코드 자동화 에이전트

| 항목 | 내용 |
|:---|:---|
| 성격 | Power 시리즈(Power Automate 등) 계보의 **에이전트·업무 자동화 제작 도구** — 대화형 에이전트와 에이전트 플로우(트리거→액션)를 로우코드/노코드로 구성 |
| 할 수 있는 것 | **단일 업무의 자동화**: 커넥터 기반 실행(메일 발송, 승인 라우팅, 파일 저장, 폼 처리), M365·Dataverse·수백 개 커넥터 연동, 간단한 대화형 에이전트 게시 |
| 할 수 없는 것 | 커넥터 범위 밖의 커스텀 로직·알고리즘, 복잡한 다중 시스템 오케스트레이션, 독립 서비스(자체 UI·API) 배포 |
| 적합 신호 | "X가 오면 Y를 자동으로 해 달라"(트리거-액션 구조), 대상 시스템이 M365·표준 커넥터 범위 |
| 부적합 신호 | 커넥터가 없는 사내 시스템 연동, 상태·로직이 복잡한 서비스, 성능·확장 요구 |

### 2.4 Databricks — 데이터 플랫폼 기반 에이전트

| 항목 | 내용 |
|:---|:---|
| 성격 | 거버넌스된 사내 데이터(레이크하우스) 위에서 동작하는 **데이터 중심 분석·에이전트 플랫폼** (Genie 자연어 질의, Agent Bricks/App Builder 로우코드 에이전트·앱) |
| 할 수 있는 것 | 자연어 데이터 질의·분석, 데이터 기반 에이전트·간단 앱을 로우코드/노코드로 제작, ML 워크로드 |
| 할 수 없는 것 | 데이터 플랫폼 밖 업무 시스템의 실행 자동화, 범용 웹 서비스 구축 |
| 적합 신호 | 핵심 가치가 **데이터**에 있는 과제 — "데이터를 물어보고 싶다/집계·예측하고 싶다/데이터 기반 판단 에이전트" |
| 부적합 신호 | 데이터보다 업무 로직·연동이 중심, 데이터가 플랫폼 밖에만 존재 |

### 2.5 Sandbox (Claude Code) — 커스텀 에이전트 풀코드 개발

| 항목 | 내용 |
|:---|:---|
| 성격 | 위 4가지로 해결되지 않는 과제를 위한 **풀코드 개발 환경** — Claude Code + 골든 패스로 에이전트를 직접 개발·배포 |
| 적합 대상 | 다중 시스템 연동, 커스텀 로직·알고리즘, 자체 UI·API 서비스, 상용 툴 조합으로 불가능한 복합 에이전트 |
| **자격 요건 (전원 충족 필수)** | ① **Green Belt 이상** ② **Claude Code 교육 이수** (VS Code 환경 활용 가능) ③ 과제가 **조직(팀 이상)의 생산성 향상** 목적 ④ **T2Y에 해당(기여)하는 에이전트**일 것 |
| 배정 안 되는 경우 | 개인용 도구, 상용 툴로 커버 가능, 자격 미충족(→ 교육 안내), T2Y 무관 과제(→ 보류·보드 회부) |

## 3. 트리아지 결정 트리

챗봇은 아래 순서로 판별한다 — **앞 단계에서 해결되면 뒤로 가지 않는다** (최소 수단 우선).

```
Q1. 수혜 범위가 개인인가?
 ├─ 예 → Q1-1. M365 데이터(메일·문서) 중심 조회·요약·초안인가?
 │        ├─ 예 → [MS Copilot 안내] (+프롬프트 세팅 가이드)
 │        └─ 아니오(사내 공통 업무·간단 챗봇) → [A.Biz 안내]
 │        ※ 개인용 커스텀 에이전트 요청 → Sandbox 미배정 원칙 고지
 └─ 아니오(팀·조직) → Q2

Q2. "트리거 → 액션"형 단일 업무 자동화인가? (대상이 M365·표준 커넥터 범위)
 ├─ 예 → [Copilot Studio 안내]
 └─ 아니오 → Q3

Q3. 과제의 핵심 가치가 데이터(질의·분석·데이터 기반 판단)인가?
 ├─ 예 → [Databricks 안내] (Genie/Agent Bricks)
 └─ 아니오 → Q4

Q4. 간단한 문서 기반 RAG 챗봇 수준인가?
 ├─ 예 → [A.Biz 안내] (노코드 RAG)
 └─ 아니오(커스텀 로직·다중 연동·서비스 배포 필요) → Q5

Q5. Sandbox 자격·목적 게이트 (전원 충족 필수)
 ├─ Green Belt 이상 + Claude Code 교육 이수?  → 미충족: [교육 안내 후 재도전] (판정: 미승인)
 ├─ 조직 생산성 향상 목적?                      → 미충족: [개인용 미배정 고지] (판정: 미승인)
 ├─ T2Y 기여 에이전트?                          → 불명확: (판정: 보류 — 보드 회부)
 └─ 전원 충족 → (판정: 승인 권고) → 보드 송부
```

## 4. 판별 질문 세트 (챗봇 표준 질문)

| # | 질문 | 판별 목적 |
|:--|:---|:---|
| 1 | 누구를 위한 것입니까 — 본인, 팀, 조직 전체? | Q1 수혜 범위 |
| 2 | 무엇이 자동으로 "실행"되기를 원합니까, 아니면 정보를 "얻고" 싶습니까? | 조회형(Copilot/A.Biz) vs 실행형(Studio/Sandbox) |
| 3 | 연결해야 하는 시스템·데이터는 무엇입니까? | 커넥터 범위(Studio) vs 데이터 플랫폼(Databricks) vs 커스텀 연동(Sandbox) |
| 4 | 반복 주기와 사용 인원은 어느 정도입니까? | 개인 프롬프트 세팅 vs 팀 서비스 |
| 5 | 기존 사내 도구로 시도해 본 적이 있습니까? 안 됐다면 무엇이 안 됐습니까? | 상용 커버리지 검증 (원칙 1) |
| 6 | 다루는 데이터에 민감정보가 포함됩니까? | 정책 적합성 (마스킹 원칙·반입 절차 안내) |
| 7 | (Sandbox 후보 시) Green Belt·Claude Code 교육 이수 여부, 과제의 T2Y 연관성은? | Q5 게이트 |

## 5. 판정 규칙 (승인 / 미승인 / 보류)

| 판정 | 조건 | 후속 |
|:---|:---|:---|
| **승인 (권고)** | Q1~Q4에서 상용 툴로 해결 불가 + Q5 게이트 전원 충족 | 근거 리포트와 함께 보드 송부 → 보드 승인 결정 시 Sandbox 배정·진입 |
| **미승인** | 상용 툴로 커버 가능(해당 툴+가이드 안내로 종결) / 개인용 / 자격 미충족(교육 안내) | 결과·사유를 신청자에게 즉시 안내, 보드에는 집계 보고 |
| **보류** | T2Y 연관성 불명확, 정책 우려(민감정보 등), 판단 근거 부족 | 근거 리포트와 함께 **보드 회부** — 보드가 추가 확인 후 결정 |

- 모든 판정은 근거(대화 요약·판별 결과)와 함께 기록되고(판정 로그), 보드 피드백으로 기준을 보정한다

## 6. 경계 사례 판정 가이드

| 사례 | 판정 |
|:---|:---|
| "Copilot Studio로 가능해 보이는데 Claude Code로 만들고 싶다" | 상용 우선(원칙 1) — Copilot Studio 안내. 시도 후 한계 확인 시 재신청 |
| "개인 업무용 에이전트를 만들고 싶다 (자격은 충족)" | 미승인 — 개인용 미배정 원칙. MS Copilot/A.Biz 안내 |
| "데이터 분석 + 커스텀 UI 서비스가 모두 필요하다" | Databricks(App Builder) 우선 검토 → 로우코드 한계 확인 시 Sandbox 후보로 승격 |
| "상용 툴 커버 여부가 애매한 복합 과제" | 보류 — 근거 리포트와 함께 보드 회부 |
| "자격은 미달인데 과제는 T2Y에 딱 맞는다" | 미승인(교육 안내) — 단, 근거 리포트에 과제 가치를 명기해 보드가 팀 구성 조정(자격자 합류)을 검토할 수 있게 함 |

## 7. 기준 운영·개정

- 본 기준은 챗봇 Knowledge Base(RAG)에 탑재되어 판정의 근거로 사용된다
- 판정 로그·보드 피드백을 분기 단위로 검토해 기준을 개정한다 (자동 승인 확대 여부 포함)
- 사내 도구의 기능 변화(예: 사내 테넌트의 Copilot 실행 기능 활성화, A.Biz 기능 확장)는 §2 프로파일에 수시 반영한다
