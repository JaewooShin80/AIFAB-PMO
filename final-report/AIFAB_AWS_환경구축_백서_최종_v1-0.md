## 서문

사내 교육으로 Claude Code·Copilot을 다루는 시티즌 개발 인력이 확보되었지만, 이들이 아이디어를 실제 서비스로 만들 수 있는 **통제된 실행 환경**은 존재하지 않았다. 통제 없는 자율 개발은 보안·데이터·비용 리스크를 수반하고, 반대로 과도한 통제는 현장의 혁신 동력을 꺼뜨린다.

AIFAB의 방향은 사내 노스폴(North Star)인 **T2Y**가 정의한다. T2Y 달성에 직결되는 과제를 **탑다운(Top-down) 과제**로 정의하고, AIFAB AWS 환경은 이 **T2Y 달성을 위한 과제와 에이전트(Agent)를 개발·지원하는 플랫폼**으로 구축되었다. 탑다운 트랙은 T2Y 공식 과제의 에이전트를 운영 수준으로 개발·배포하고, 바텀업 샌드박스는 현장의 자율 아이디어를 수용해 검증한 뒤 우수 과제를 격상시켜 T2Y로 수렴시킨다 — 두 트랙 모두 종착점은 T2Y다.

AIFAB AWS 환경은 자율과 통제의 딜레마에 대한 답으로 설계되었다. 핵심 명제는 하나다 — **"계정 안에서는 자율, 조직의 금지선은 강제."** 정책을 문서가 아닌 플랫폼 차원에서 강제하고, AI가 심사와 모니터링을 자동화하며, 사람(AI Board)은 판단이 필요한 지점에만 개입한다.

본 백서는 이 환경의 설계 철학, 아키텍처, 구축 과정을 기록한다.

## 1. 설계 철학 — 5원칙

| # | 원칙 | 구현 방식 |
|:--|:---|:---|
| 1 | **Security by Design** | 보안을 절차가 아닌 아키텍처에 내재화 — 민감정보 마스킹 원칙(샌드박스), 사내망 한정, SCP 가드레일은 계정 내 누구도 해제 불가 |
| 2 | **계정 격리 + 조직 가드레일** | 과제별 AWS 계정 분리로 장애·부하·보안 이슈의 전파 차단, OU 단위 SCP로 금지선 일괄 상속 |
| 3 | **전 생애주기 자동화** | 신청 → AI 심사 → 승인 → 프로비저닝(30분) → 개발·배포 → 모니터링 → 격상/선셋까지 사람 개입 최소화 |
| 4 | **"만들기 전에, 있는 것부터"** | AI 트리아지가 기존 사내 도구로 해결되는 과제를 걸러 도구를 안내 — 신규 개발이 꼭 필요한 과제에만 자원 집중 |
| 5 | **비용의 구조적 통제** | 시간제 가동 + 자동 선셋 + 쿼터의 3중 장치 — 절약을 운영자의 노력이 아닌 구조로 보장 |

각 원칙의 선언(규범)과 상세 근거는 아래와 같다. 문서 표기는 약어를 사용하며 전체 경로는 부록을 따른다 — **바텀업 v1-0**(바텀업 샌드박스 기획안), **상세 v2-1**(샌드박스 상세 기획안), **탑다운 v1-0**(탑다운 기획안), **격상 표준**(격상과제 이관·수용 절차 표준). 하위 문서에 명시가 없는 선언은 *(본 백서 신규 선언)* 으로 표기한다.

### 1.1 원칙 1 — Security by Design

**선언**
- 보안 통제는 문서·절차가 아니라 아키텍처로 구현한다. 사람이 지키는 규칙은 어겨질 수 있지만, 구조가 강제하는 규칙은 어길 수 없다. *(본 백서 신규 선언)*
- 실데이터 사용은 허용되나 **민감정보는 마스킹하여 사용하는 것을 원칙**으로 한다 — Macie가 마스킹되지 않은 민감정보 유입을 상시 탐지한다. (바텀업 v1-0 §4 — 본 백서에서 개정 후 전 문서 반영 완료, 정보보호팀 승인은 협의 항목 B-6)
- 모든 접근은 사내망으로 한정하고 인터넷 인바운드를 차단한다. (상세 v2-1 4장)
- 가드레일(SCP)은 계정 내 관리자 권한으로도 해제할 수 없다. (바텀업 v1-0 §3.1)

**상세 근거**

| 내용 | 문서 위치 |
|:---|:---|
| 데이터 반입 정책(민감정보 마스킹 원칙)·Macie 상시 탐지·산출물 유출 통제 | 바텀업 v1-0 §4 |
| 보안 4계층 설계·KMS CMK 전략·Permission Boundary·Secrets Manager | 상세 v2-1 5장 (5.2~5.4) |
| 탑다운 실데이터 보호 SCP 3종·데이터 등급 분류 | 탑다운 v1-0 3.1·4.1.1 |
| Bedrock 거버넌스 (Guardrails·호출 로깅) | 상세 v2-1 6.3 |
| 본 백서 상세 | 6장·7장 |

### 1.2 원칙 2 — 계정 격리 + 조직 가드레일

**선언**
- "계정 안에서는 자율, 조직의 금지선은 강제" — 개발자에게 넓은 자율성을 주되, 격리 전제는 계정 안에서 누구도 변경할 수 없다. (바텀업 v1-0 §3.1)
- 과제 하나의 실험 실패가 다른 과제·공식 환경에 영향을 주지 않는 구조를 계정 경계로 보장한다. *(본 백서 신규 선언 — 바텀업 v1-0 §11 기대 효과에서 도출)*

**상세 근거**

| 내용 | 문서 위치 |
|:---|:---|
| OU 구조(Shared Services/Top-down/Sandbox)·SCP 가드레일 6종 | 상세 v2-1 3.2 (3.2.2) |
| 계정 벤딩 자동화·선셋 시 격리·폐기 절차 | 상세 v2-1 3.2.1 |
| 본 백서 상세 | 3장 |

### 1.3 원칙 3 — 전 생애주기 자동화

**선언**
- 사람의 개입은 판단이 필요한 지점(HITL — 보드 승인·게이트 심사)으로 한정하고, 나머지는 플랫폼이 수행한다. *(본 백서 신규 선언 — 바텀업 v1-0 §5 HITL 조항에서 도출)*
- 환경 준비는 신청 승인 후 30분 이내, AI 심사는 10분 이내를 목표로 한다. (바텀업 v1-0 §11, 상세 v2-1 6.1.1)
- 인프라 변경은 Terraform 코드 리뷰(PR)로만 반영한다 — 수기 변경 경로를 두지 않는다. (바텀업 v1-0 §6)

**상세 근거**

| 내용 | 문서 위치 |
|:---|:---|
| 전 생애주기 적용 범위 (신청→격상/선셋) | 바텀업 v1-0 §1.3 |
| 심사 파이프라인 6단계·Terraform 모듈·state 운영 | 상세 v2-1 6.1·3.5 |
| 인증·권한 자동 체인 (Entra ID→SCIM→권한세트) | 상세 v2-1 3.3, 바텀업 v1-0 §3.2 |
| 격상·선셋 자동화 | 바텀업 v1-0 §7, 격상 표준 전체 |
| 본 백서 상세 | 4장·8장·10장·11장 |

### 1.4 원칙 4 — "만들기 전에, 있는 것부터"

**선언**
- 기존 사내 AI 도구로 해결되는 과제는 만들지 않는다 — 도구와 활용 가이드를 안내하고 샌드박스를 배정하지 않는다. (상세 v2-1 6.1, 바텀업 v1-0 §1.2·§5)
- 심사 에이전트는 권고만 하며, 반려 권한은 AI Board에만 있다. (상세 v2-1 6.1)

**상세 근거**

| 내용 | 문서 위치 |
|:---|:---|
| 트리아지 4분기 (도구 안내/배정/역량 준비/보드 회부) | 바텀업 v1-0 §5 |
| 다각도 판단 기준·대화형 보완·Knowledge Base | 상세 v2-1 6.1 |
| 저위험 자동 승인 단계 확대 | 바텀업 v1-0 §5, 상세 v2-1 10장(4단계) |
| 본 백서 상세 | 8장 |

### 1.5 원칙 5 — 비용의 구조적 통제

**선언**
- 비용 절약은 운영자의 노력이 아니라 구조(시간제 가동·자동 선셋·쿼터)로 보장한다. (바텀업 v1-0 §10 "3중 구조")
- 운영급(상시) 가동은 격상을 통해서만 부여한다 — 상시 가용성 필요 자체가 격상 사유다. (바텀업 v1-0 §3.3)
- 태그 없는 자원은 존재할 수 없다 — 표준 태그를 정책으로 강제하고 계정별 예산 알람을 상시 가동한다. *(본 백서 신규 선언 — 탑다운 v1-0 9.4에서 도출)*

## 2. 전체 아키텍처

AIFAB 환경은 하나의 Landing Zone 위에서 운영 등급이 다른 **두 트랙**으로 구성된다.

![그림 1. AWS 전체 아키텍처](../fab/media/fig1-aws-architecture.png)

*그림 1. AWS 전체 아키텍처 — 계정 분리 및 신청·프로비저닝·개발 흐름*

| 구분 | 바텀업 샌드박스 | 탑다운 (공식 과제) |
|:---|:---|:---|
| 과제 성격 | 현장 자율 제안 (실험·검증) | 사업 계획 확정 과제 |
| 심사 | AI 다각도 심사·트리아지 → 보드 승인 | AI 심사 바이패스(적합성 심사 한정), 보드 확인 |
| 실행 환경 | 개발과 동일 환경에 dev 배포 | 스테이징·운영 분리 |
| 가동 | 시간제 (셧다운/웨이크업) | 상시 (서비스 수준 관리) |
| 배포 | dev까지 (게이트 없음) | 스테이징 → 승인 게이트 → 운영 |
| 데이터 | **실데이터 허용 — 민감정보 마스킹 원칙** | 통제 하 실데이터 허용 (등급 심사, 민감정보 원본은 심사 후 허용) |
| 종료 | 저활용 자동 선셋 | 사업 판단 종료·이관 |

우수한 바텀업 과제는 **격상 경로**(10장)를 통해 재작성 없이 탑다운 운영 환경으로 이관된다 — 현장 혁신이 전사 서비스로 확산되는 파이프라인이다.

## 3. 계정·거버넌스

- **OU 구조**: Root 아래 Shared Services(공통 자동화·조직 로그) / Top-down / Bottom-up Sandbox OU로 분리. 과제 승인 시 해당 OU 아래 계정이 자동 생성된다
- **SCP 가드레일 6종** (Sandbox OU 일괄 상속): 리전 제한, 인터넷 경로 생성 금지, 네트워크 변경 제한, 감사·보안 서비스 보호, S3 퍼블릭 노출 금지, 조직 이탈 금지. 탑다운 OU에는 실데이터 보호 3종(KMS 키 단기 삭제 금지·Macie 비활성화 금지·CloudTrail 중지 금지)을 추가 적용
- **계정 벤딩·회수**: Organizations API + Lambda 파이프라인으로 계정 발급을 자동화하고, 선셋 시 Quarantine OU 격리 → 90일 후 폐기의 역방향 절차까지 정의

## 4. 인증·권한

- **자동 권한 체인**: 승인 완료 → Entra ID 과제 그룹 추가(사번 기반) → SCIM 동기화 → IAM Identity Center 권한세트 할당 → 액세스 포털 접근. 입사·퇴사·부서이동이 AWS 접근 권한에 자동 반영되며, Orphan 계정(담당자 이탈)은 Entra ID 비활성화 이벤트로 자동 탐지된다
- **최소권한 이중 장치**: 역할별 권한세트(개발자·인프라팀·정보보호팀·보드·멘토별 차등) + **IAM Permission Boundary** — 개발자가 자신보다 넓은 권한의 역할을 만들 수 없도록 권한 상승을 원천 차단

## 5. 네트워크

![그림 2. 네트워크(VPC/Subnet) 구성도](../fab/media/fig2-network-vpc.png)

*그림 2. 네트워크 구성 — 사내망 전용, 내부 ALB(Private Subnet), VPC Endpoint 통신*

- **사내망 한정**: 인터넷 인바운드 차단, Direct Connect/VPN + Transit Gateway 경유 접근만 허용. 아웃바운드는 화이트리스트 기반 NAT 통제
- **AWS 서비스 통신은 VPC Endpoint 경유**: ECR·STS·CloudWatch Logs·Bedrock-runtime·Secrets Manager 등 인터페이스 엔드포인트 + S3·DynamoDB 게이트웨이 엔드포인트 — 서비스 트래픽이 인터넷을 거치지 않음
- **대역·DNS 중앙 관리**: VPC IPAM으로 조직 CIDR 풀을 중앙 관리해 계정 증가 시 대역 충돌을 방지하고, Route 53 Resolver로 온프렘 DNS와 양방향 연동

## 6. 보안 아키텍처

![그림 3. 보안 아키텍처 (4계층 + 통합 탐지)](../fab/media/fig3-security-architecture.png)

*그림 3. 보안 4계층 — 계정·접근 / 네트워크 / 데이터 / 탐지·감사*

- **데이터 정책**: 샌드박스는 실데이터 사용을 허용하되 **민감정보 마스킹을 원칙**으로 한다 — 민감정보(개인정보 등)는 마스킹·비식별 처리 후 반입하며, **Macie가 마스킹되지 않은 민감정보 유입을 상시 탐지**해 정책의 실효성을 기술적으로 담보한다. 탑다운은 데이터 등급 분류(공개/내부/기밀/극비) 기반 정보보호팀 사전 심사를 거쳐 민감정보 원본까지 통제 하에 허용
- **암호화**: 계정별 CMK(고객 관리 키) 원칙 — S3·RDS·EBS 전 구간 암호화, 크로스어카운트 키 사용은 Terraform 실행 롤로 한정
- **시크릿**: Secrets Manager 일원화 — RDS 자격증명 30일 자동 로테이션, 컨테이너 런타임 주입, 파이프라인에서 하드코딩 검출 시 자동 차단
- **탐지·감사**: 전 계정 CloudTrail 전수 기록, GuardDuty·Security Hub 통합, 위반 시 보드·정보보호팀 SNS 알림. Critical Finding은 4시간 내 정보보호팀 통보
- 산출물 외부 유출 통제: 다운로드 금지·외부 공유 차단

## 8. 신청·심사 자동화 — AI 트리아지

![그림 5. AI Agent 아키텍처](../fab/media/fig4-ai-agent.png)

*그림 5. 심사 에이전트(다각도 심사·트리아지) + 주간 리포트 에이전트*

심사 에이전트(Bedrock Claude)는 신청 과제를 데이터 현황·업무 맥락·활용 범위·정책 적합성의 다각도로 분석하고, 부족한 정보는 대화형 질문으로 보완한 뒤 **4가지 경로로 트리아지**한다.

| 판정 | 처리 |
|:---|:---|
| ① 기존 도구 안내 | 사내 상용 AI 도구(A.Biz·Copilot·Databricks 등)로 해결 가능 → 도구 + 맞춤 가이드 안내, 샌드박스 미배정 |
| ② 샌드박스 배정 | 신규 에이전트 개발 필요 + 신청자 역량 요건 충족 → 승인 절차 진행 |
| ③ 역량 준비 안내 | 요건 미충족 → 교육 과정 안내 후 재신청 유도 |
| ④ 보드 검토 회부 | 판단 애매·정책 우려 → 근거 리포트와 함께 회부 |

최종 승인 권한은 AI Board가 보유한다(Human-in-the-Loop). 에이전트는 권고만 하며, 정확도 검증 후 저위험 건부터 자동 승인을 단계적으로 확대한다.

## 9. 개발환경과 CI/CD

### 9.1 웹 IDE — 브라우저 로그인 한 번으로 개발 시작

개발환경은 **Coder(OSS) 플랫폼 + EC2 워크스페이스**로 구축한다 (SageMaker Code Editor와 비교 검토 후 확정 — GitLab 중심 형상관리 연계·하네스 전면 통제·확대 시 비용 우위).

- SSO 로그인 → 브라우저 VS Code 즉시 진입 — 설치 0건, API 키 0건
- 골든 AMI에 Claude Code + AIFAB 하네스(전역 CLAUDE.md·commands·plugins) 선탑재 — 첫 진입 즉시 `claude` 사용 가능
- 유휴 자동 종료(autostop)로 시간제 가동 정책과 연동

### 9.2 골든 패스 — 폼 입력에서 배포까지

과제 승인 시 Terraform이 워크스페이스 + GitLab 레포(스켈레톤) + CI/CD 파이프라인을 자동 생성한다. 표준 템플릿 7종: ECS on Fargate·Lambda·RDS·비용태그·관측성·Secrets Manager·Bedrock 연동.

![그림 7. CI/CD 및 배포 구조](../fab/media/fig5-cicd.png)

*그림 7. 바텀업 CI/CD — dev 배포까지 자동, 운영급 배포는 격상 후 탑다운*

- 파이프라인: GitLab 머지 → CodePipeline → CodeBuild(빌드·테스트·SAST(Semgrep)·시크릿 검출(gitleaks)) → ECR(Tag 불변) → Inspector 스캔 → **통과 시에만** Fargate dev 배포
- 이미지 서명 검증 — 서명되지 않은 이미지의 배포 시도는 게이트에서 자동 차단
- 형상(Git)과 실행(컨테이너)의 분리 — 격상 시 재작성 없는 이관의 전제 조건
- 인프라 변경은 Terraform 코드 리뷰(PR) 기반으로만 반영. state는 S3+DynamoDB 잠금, 모듈은 SemVer 버전 관리, 주 1회 drift 탐지

## 10. 가동·비용·생애주기

- **시간제 가동**: EventBridge Scheduler가 태그 기준 일괄 셧다운/웨이크업 (기본 평일 08:00~20:00, 시간 외 포털 연장 신청 — 저위험 자동 승인). 상시 가용성이 필요해진 과제는 그 자체가 격상 사유
- **자동 선셋**: 4주 무접속 또는 활성 기준 미달 4주 → 알람 → 2주 유예 → 차단 → 4주 보관 후 회수·파기. Orphan 과제는 즉시 선셋 후보
- **쿼터·예산**: 과제별 컴퓨팅·스토리지·API 쿼터 + Rate Limit, 표준 태그 4종 강제(Tag Policy), 계정별 Budgets 80%/100% 알람
- **비용 실적(추정)**: 시간제 가동(가동률 약 36%) 적용 시 과제 10~15개 규모 **월 약 750~850 USD** — 상시 가동 대비 컴퓨팅 약 60% 절감

## 11. 격상 — 현장 아이디어의 전사 확산 경로

- 판정 기준: 활성 사용자·사용 빈도·업무 효과·안정성 또는 상시 가용성 필요 — 분기 보드 심의
- 이관 표준: 7단계 체크리스트(코드 리뷰 → 문서화 → 데이터 연동 재설계(마스킹 해제)·보안 재심사 → 운영 조직 지정 → 재프로비저닝 → Git 이관·검증 → 원 환경 정리), 수용 기준(SAST Critical/High 0건·시크릿 0건·커버리지 70% 기본안), 하이퍼케어 2주
- 계정 간 이관 기술 절차(S3 복제·RDS 스냅샷 KMS 재암호화·ECR 복사·Secrets 재생성·DNS 전환)까지 표준화 — 이관 시점의 임기응변을 배제

## 12. 운영·관측

- 주간 리포트 에이전트: 신청·승인 현황, 과제별 사용량, 우수과제·선셋 후보, 보안 이벤트를 자동 집계·요약해 보드·경영진 보고
- 운영 KPI: 신청 건수, 승인율(트리아지 분기별), 활성 과제 수, 격상 건수, 절감 공수, 평균 심사 소요, 선셋 회수 비용
- 패치(SSM Patch Manager 주간 자동), 런북 기반 장애 대응, 스프린트 기간 온콜(초동 30분)

## 14. 맺음말 — 기대 효과와 향후 과제

**기대 효과**
- 심사 리드타임: 수작업 수 일 → AI 심사 10분 + 보드 확인 1일 이내
- 환경 준비: 수 주 → 자동 프로비저닝 30분 이내
- 트리아지로 불필요한 환경 배정 원천 차단 — 자원을 신규 개발이 필요한 과제에 집중
- 계정 격리·가드레일로 실험의 리스크가 공식 환경에 전파되지 않는 구조
- 격상 파이프라인으로 검증된 현장 아이디어의 재작성 없는 전사 확산

**향후 과제 (v2-0 개정 시 반영)**
- 파일럿 실측 결과(KPI·비용·심사 정확도) 및 협의 확정 수치 반영
- 저위험 자동 승인 확대 기준, 코호트 확대(5~10팀) 운영 실적
- 지속 사이클 운영 데이터 기반의 플랫폼 개선 이력
