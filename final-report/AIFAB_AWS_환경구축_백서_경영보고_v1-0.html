Philosophy

배경과 설계 철학 두 트랙 모두 종착점은 T2Y

핵심 명제. "계정 안에서는 자율, 조직의 금지선은 강제" — 사내 노스폴 T2Y 달성을 위한 과제·에이전트를 개발·지원하는 플랫폼. 정책을 문서가 아닌 플랫폼 차원에서 강제하고, AI가 심사·모니터링을 자동화하며, 사람(AI Board)은 판단 지점에만 개입한다.
원칙구현 방식
1. Security by Design보안을 아키텍처에 내재화 — 민감정보 마스킹 원칙, 사내망 한정, 해제 불가 SCP 가드레일
2. 계정 격리 + 가드레일과제별 AWS 계정 분리로 리스크 전파 차단, OU 단위 금지선 일괄 상속
3. 전 생애주기 자동화신청 → AI 심사 → 프로비저닝(30분) → 개발·배포 → 격상/선셋까지 사람 개입 최소화
4. "만들기 전에, 있는 것부터"가이드 챗봇이 기존 도구로 해결되는 과제를 걸러 안내 — 꼭 필요한 개발에만 자원 집중
5. 비용의 구조적 통제시간제 가동 + 자동 선셋 + 쿼터 3중 장치 — 절약을 노력이 아닌 구조로 보장

Governance & Security

거버넌스·보안 규칙이 아니라 구조로 강제

조직 가드레일 (SCP) — 관리자도 해제 불가

  • 공통 6종: 리전 제한·인터넷 경로 생성 금지·네트워크 변경 제한·감사 보호·S3 퍼블릭 금지·조직 이탈 금지
  • 탑다운 +3종: 암호화 키 삭제·탐지 해제·감사 중단 원천 차단
  • 예외는 IaC 자동화 경로(코드 리뷰) 단 하나
  • 계정 발급 30분 자동화 ↔ 선셋 시 격리 → 90일 후 폐기

4계층 심층 방어 + 데이터 정책

  • 계정·접근 / 네트워크 / 데이터 / 탐지·감사 — 탐지 결과는 Security Hub 단일 창구
  • 민감정보 마스킹 원칙 — Macie가 위반 상시 탐지
  • 암호화 키 계정별 분리(CMK), 시크릿 하드코딩은 파이프라인이 차단
  • AWS 접근에 비밀번호 없음 — 사내 SSO·임시 자격증명만

AI Governance

AI 거버넌스 LLM 호출도 통제 대상 자원

거버넌스 질문장치
누가 쓸 수 있나키리스(IAM 롤) + 승인 모델 화이트리스트 — API 키를 만들지도 배포하지도 않음
무엇을 주고받나Guardrails — 민감정보(PII) 필터·금지 주제·콘텐츠 필터, 사용자 실수도 플랫폼이 차단
무엇을 했는지 남나프롬프트·응답 전량 기록 — 전용 암호화 키, 감사 역할만 열람 (사고 조사·책임 추적)
얼마나 쓸 수 있나과제별 토큰 지표·예산 상한 알람 — 비용 사고를 "월말 발견"이 아닌 "당일 알람"으로
LLM 백엔드 이원화. 코딩 작업 = Claude 팀플랜(개인 시트 로그인, 정보보호팀 승인 전제) / 에이전트 호출 = Bedrock(키리스·전량 로깅·Guardrails). Bedrock은 대체 경로로 병행 유지 — 승인 지연 시 즉시 전환 가능.
