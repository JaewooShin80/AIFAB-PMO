Why

추진 배경 및 목적

  • 교육으로 확보한 시티즌 개발인력이 있으나, 탑다운(AIFAB) 범위 밖 현장 개선 아이디어를 수행할 환경이 부재
  • 통제 없는 자율 개발은 보안·데이터·자원 리스크 — 정책이 플랫폼 차원에서 강제되는 샌드박스가 전제 조건
  • AI 에이전트가 심사·모니터링·리포트를 자동화하여 최소 인력으로 지속 운영
  • 우수과제는 탑다운 격상 경로로 전사 확산 (탑다운 기획안 연계)

Triage

AI 심사·트리아지 — "만들기 전에, 있는 것부터"

  • 신청 과제를 다각도 분석(데이터 현황·업무 맥락·활용 범위·정책·중복·자원) 후 최적 수행 수단으로 안내
  • 사내 AI 도구 카탈로그: A.Biz · MS Copilot · Copilot Studio · Databricks · Claude Code
판정 ①기존 도구 안내상용 도구로 가능 시 맞춤 활용 가이드 반환 — 샌드박스 미배정
판정 ②샌드박스 배정신규 개발 필요 + Claude Code 적합 + 교육 이수·경험 보유 시에만
판정 ③역량 준비 안내요건 미충족 시 사내 교육 안내 후 재신청 유도
판정 ④보드 검토판단 애매·정책 우려 시 근거 리포트와 함께 회부
효과. 기존 도구로 해결되는 과제는 환경 배정 없이 종결 — 자원과 운영 부담이 신규 개발이 꼭 필요한 과제에만 집중된다. 최종 승인 권한은 AI Board 보유(HITL).

Process

운영 프로세스 — 신청부터 격상·선셋까지 자동화

  • 신청 → AI 심사(대화형 보완) → 트리아지 분기 → 보드 승인 → 자동 프로비저닝 → 개발·dev 배포
  • 선셋: 4주 미사용 시 알람 → 2주 유예 → 자원 회수·데이터 파기 (오너 소명 시 연장)
  • Orphan 과제: 담당자 퇴직·이동을 인사 연동(Entra ID)으로 자동 탐지 → 즉시 이관·종료 처리
  • 격상: 성과 기준 충족 또는 상시 가용성 필요 시 분기 보드 심의로 탑다운 전환

5
Architecture & Security

격리와 가드레일 — 안에서는 자율, 금지선은 강제

  • 과제별 계정 격리 — 장애·부하·보안 이슈가 공식(탑다운) 환경에 영향 없음
  • SCP 가드레일 6종: 리전 제한, 인터넷 경로 생성 금지, 네트워크 변경 제한, 감사 보호, 퍼블릭 노출 금지, 조직 이탈 금지 — 계정 안의 누구도 우회 불가
  • 암호화 키 계정별 분리 관리 + 개발자 권한 상승 원천 차단(권한 경계) — 과제 간 데이터 혼용 불가 구조
  • AI 호출 전량 감사 기록 + 민감정보 필터(가드레일) + 팀별 사용량 상한 — 무제한 AI 과금 리스크 차단
  • 사내망 한정 접근(인터넷 인바운드 차단), 사번 기반 SSO 자동 연동
  • 마스킹되지 않은 민감정보 유입은 Macie가 상시 탐지, 산출물 다운로드 금지(DRM·워터마크)
  • 인프라 변경은 Terraform 코드 리뷰(PR)로만 — 콘솔 우회 경로 차단

Ask

기대 효과 및 의사결정 요청 사항

기대 효과

  • 심사 수일 → 1일 이내, 환경 준비 수주 → 30분
  • 트리아지로 불필요한 환경 배정 원천 차단
  • 시간제 가동·자동 선셋으로 유휴 비용 구조적 통제
  • 계정 격리·SCP로 공식 환경 무영향 보장
  • 격상 파이프라인으로 현장 혁신의 전사 확산
