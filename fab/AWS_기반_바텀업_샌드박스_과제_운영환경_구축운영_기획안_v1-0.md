# AWS 기반 바텀업(샌드박스) 과제 운영환경 구축·운영 기획안

AI SAND BOX : 현장 주도(Bottom-up) 과제 지원 플랫폼

- 문서 구분 : 운영 기획(안)
- 버전 : v1.0
- 작성일 : 2026. 07. 11.
- 작성자 : 신재우
- 관련 문서 : AWS 기반 샌드박스 구축·운영 기획안 v2-1 (플랫폼 상세, 이하 "상세 기획안"), AWS 기반 탑다운(AIFAB) 과제 운영환경 구축·운영 기획안 v1-0 (이하 "탑다운 기획안")

---

## 1. 추진 배경 및 목적

### 1.1 추진 배경

- 사내 교육을 통해 클로드 코드·코파일럿을 다루는 시티즌 개발인력이 확보되었으나, 탑다운(AIFAB) 과제 범위에 들지 않는 현장의 개선 아이디어를 수행할 환경과 방법론이 부재함
- AIFAB을 위해 마련한 샌드박스를 확대하여, 시티즌 개발자가 자율(바텀업) 과제를 수행할 수 있는 통제된 실험 환경을 제공하고자 함
- 통제 없는 자율 개발은 보안·데이터·자원 리스크를 수반하므로, 정책이 플랫폼 차원에서 강제되고 AI가 심사·모니터링을 자동화하는 운영 체계가 전제 조건임
- 우수 과제는 탑다운으로 격상하는 경로(탑다운 기획안 6장)를 통해 현장 혁신을 전사로 확산함

### 1.2 목적

- 시티즌 개발자에게 계정 수준으로 격리된 dev급 실험 환경을 신청부터 선셋까지 전 생애주기 자동화로 제공
- AI 심사 에이전트의 다각도 심사·트리아지로 "만들기 전에, 있는 것부터" — 기존 사내 AI 도구로 해결되는 과제는 도구를 안내하고, 신규 개발이 필요한 과제만 샌드박스에 배정
- 시간제 가동·자동 선셋·쿼터로 비용을 구조적으로 통제
- Security by Design: 실데이터 금지, 사내망 한정, 조직 차원 가드레일(SCP)을 아키텍처에 내재화

### 1.3 적용 범위

과제 신청 → AI 다각도 심사·트리아지 → 보드 승인 → 자원 프로비저닝 → 개발·dev 배포 → 운영 모니터링 → 격상 또는 선셋에 이르는 바텀업 과제 전 생애주기를 대상으로 한다.

## 2. 탑다운과의 차이 (요약)

두 환경은 같은 플랫폼 기반(Landing Zone) 위에서 운영 등급만 달리한다.

| **구분** | **바텀업 (본 기획)** | **탑다운 (탑다운 기획안)** |
|:---|:---|:---|
| 과제 성격 | 현장 자율 제안 과제 (실험·검증) | 사업 계획으로 확정된 공식 과제 |
| 심사 | AI 다각도 심사·트리아지 후 보드 승인 | AI 사전심사 바이패스, 보드 확인 |
| 실행 환경 | 개발과 동일 환경에 dev 배포 | 별도 운영 환경, 스테이징·운영 분리 |
| 가동 | 시간제 가동 (셧다운/웨이크업) | 상시 가동 (서비스 수준 관리) |
| 배포 | dev 배포까지 (승인 게이트 없음) | 스테이징 → 운영 + 승인 게이트 |
| 데이터 | 실데이터 금지 (마스킹·비식별만) | 통제 하 실데이터 연동 허용 |
| 수행 주체 | 시티즌 개발자 (교육 이수·경험 요건) | 지정된 개발·운영 조직 |
| 종료 | 저활용 자동 선셋 | 사업 판단에 따른 종료·이관 |

## 3. 계정·환경 구조

### 3.1 조직(OU)과 가드레일

상세 기획안 3.2의 조직 구조를 공유하며, 본 기획의 대상은 Bottom-up Sandbox OU다.

![그림 2. AWS 전체 아키텍처 (계정 분리 및 신청·프로비저닝·개발 흐름)](media/fig1-aws-architecture.png)

*그림 2. AWS 전체 아키텍처 (계정 분리 및 신청·프로비저닝·개발 흐름, 상세 기획안과 공용)*

- 과제 승인 시 Bottom-up Sandbox OU 아래에 과제별 계정이 자동 생성되고, OU에 부착된 SCP 가드레일 6종(리전 제한, 인터넷 경로 생성 금지, 네트워크 변경 제한, 감사·보안 서비스 보호, S3 퍼블릭 노출 금지, 조직 이탈 금지 — 상세 기획안 3.2.2)을 즉시 상속받는다
- 원칙은 "계정 안에서는 자율, 조직의 금지선은 강제" — 사용자에게 넓은 개발 자율성을 주되, 격리 전제(사내망 한정·아웃바운드 통제)는 계정 안에서 누구도 변경할 수 없다

### 3.2 인증·권한

상세 기획안 3.3(Identity Center · Entra ID 연동)을 그대로 사용한다. 승인 완료 → Entra ID 과제 그룹 추가(사번 기반) → SCIM 동기화 → SandboxDeveloper 권한세트 할당 → 액세스 포털 접근의 자동 체인으로, 입사·퇴사·부서이동이 AWS 접근 권한에 자동 반영된다.

### 3.3 실행 환경과 가동 정책

- 표준 스택: Terraform `sandbox-project` 모듈 1종 — ECS/Fargate(과제 앱 dev 배포), EC2(개발 서버), S3, RDS, 태그·예산 알람 (상세 기획안 3.5)
- **시간제 가동**: EventBridge Scheduler가 태그 기준으로 일괄 셧다운/웨이크업. 기본 평일 08:00~20:00(보드 확정), 시간 외 필요 시 포털에서 연장·즉시 웨이크업 신청(저위험 자동 승인)
- 상시 가용성이 필요해진 과제는 그 자체가 격상 사유가 된다 — 운영급 환경은 격상을 통해서만 부여 (상세 기획안 3.6)

## 4. 네트워크·보안

- 사내망 한정 접근(인터넷 인바운드 차단), Direct Connect/VPN + Transit Gateway 경유, VPC Endpoint 통신, 아웃바운드 화이트리스트 (상세 기획안 4·5장)
- **실데이터 금지**가 바텀업 데이터 정책의 근간: 마스킹·비식별·합성 데이터 또는 승인된 샘플만 반입 가능하며, 반입 시 등급 분류·승인 절차를 거친다. Macie가 실데이터·개인정보 유입을 상시 탐지하여 정책의 실효성을 기술적으로 담보
- 산출물 다운로드 금지(외부 공유 차단, DRM·워터마크), 전 계정 CloudTrail 기록과 Security Hub 통합 알림은 조직 공통 계층으로 적용

## 5. 신청·심사 (AI 다각도 심사·트리아지)

심사 에이전트(Bedrock Claude)가 신청 과제를 다각도로 분석해 가장 적합한 수행 수단으로 안내한다 (상세 기획안 6.1).

- **다각도 판단**: 데이터 현황(민감정보 포함 여부, 보관·관리 방식, AI 활용 가능한 정리 상태), 업무 맥락(현행 프로세스·목적·기대 개선 효과), 활용 범위(조직 공용 vs 개인 생산성), 정책 적합성·중복성·자원 적정성. 신청서로 부족한 내용은 대화형 질문으로 보완
- **트리아지 4분기**:

| **판정** | **처리** |
|:---|:---|
| ① 기존 도구 안내 | 사내 상용 AI 도구(A.Biz, MS Copilot, Copilot Studio, Databricks, Claude Code)로 해결 가능 시 도구 + 맞춤 활용 가이드 반환, 샌드박스 미배정 |
| ② 샌드박스 배정 | 신규 에이전트 개발 필요 + Claude Code 적합 + 신청자가 교육 이수·에이전트 작성 경험 보유 시에만 승인 절차 진행 |
| ③ 역량 준비 안내 | 요건 미충족 시 사내 교육 과정 안내 후 재신청 유도 |
| ④ 보드 검토 | 판단 애매·정책 우려 시 근거 리포트와 함께 회부 |

- 최종 승인 권한은 AI Board가 보유(HITL)하며, 에이전트는 권고만 한다. 정확도 검증 후 저위험 건부터 자동 승인을 단계 확대

## 6. 개발·배포

- 개발: EC2 개발 서버(작업 상태 유지, 자유로운 도구 설치)에서 개발하고 Git으로 형상 관리
- 배포: CodePipeline이 빌드 → 테스트·시크릿 검출 → 이미지 스캔(Inspector, 실패 시 자동 차단) → **샌드박스 내 Fargate에 dev 배포**까지 자동 수행. 바텀업에는 운영급 배포 단계와 승인 게이트가 없다
- dev 배포도 컨테이너·파이프라인 경유 원칙을 유지하여 형상(Git)과 실행(컨테이너)을 분리 — 격상 시 재작성 없는 이관의 전제 조건
- 인프라 변경은 Terraform 코드 리뷰(PR) 기반으로만 반영

## 7. 격상·선셋

- **격상**: 활성 사용자·사용 빈도·업무 효과·안정성 기준 충족 또는 상시 가용성 필요 시 분기 보드 심의로 탑다운 전환. 이관 절차(코드 리뷰, 문서화, 실데이터 연동 재설계, 운영 조직 지정, 재프로비저닝)는 탑다운 기획안 6장을 따른다
- **선셋**: 최근 4주 접속 0건 또는 활성 사용자 기준 미달 4주 지속 시 1차 알람 → 2주 유예 → 접속 차단 → 4주 보관 후 자원 회수·데이터 파기. 오너 소명 시 보드 판단으로 연장
- **Orphan 과제**: 담당자 퇴직·부서이동(Entra ID 비활성화 이벤트로 자동 탐지) 시 즉시 선셋 후보로 분류하여 이관 또는 종료

## 8. 운영·모니터링

- 과제별 쿼터(컴퓨팅·스토리지·API 호출량)와 Rate Limit, 임계치 초과 시 자동 알람·차단
- 주간 리포트 에이전트가 신청·승인 현황, 과제별 사용량, 우수과제·선셋 후보, 보안 이벤트를 자동 집계·요약하여 보드·경영진에 보고 (상세 기획안 6.2)
- 운영 KPI: 신청 건수, 승인율(트리아지 분기별), 활성 과제 수, 격상 건수, 절감 공수, 평균 심사 소요, 선셋 회수 비용

## 9. 단계별 구축 로드맵

공통 Landing Zone은 탑다운 기획안과 공동 구축(1단계)하며, 바텀업 증분 작업은 다음과 같다.

| **단계** | **기간(예시)** | **주요 작업** | **산출물** |
|:---|:---|:---|:---|
| 1단계: 공통 기반 (공동) | M1 ~ M2 | Landing Zone: Organizations·SCP, Identity Center, 네트워크, 공통 보안 | 격리된 계정 기반 환경 |
| 2단계: 신청 자동화 | M2 ~ M3 | 신청 포털, 심사 워크플로, `sandbox-project` Terraform 모듈, 쿼터·태깅, 가동 스케줄러(셧다운/웨이크업) | 신청부터 자원 생성까지 자동화 |
| 3단계: AI Agent·CI/CD | M3 ~ M5 | 심사 에이전트(다각도 심사·트리아지)와 Knowledge Base, 주간 리포트 에이전트, 표준 파이프라인 | 자동 심사·리포트, dev 배포 체계 |
| 4단계: 정착·고도화 | M5 ~ M6 | 격상·선셋 자동화, 저위험 자동 승인 확대, 비용 최적화 | 전 생애주기 운영 정착 |

## 10. 예상 비용

시간제 가동(평일 08:00~20:00, 가동률 약 36%) 적용, 과제 10~15개 규모 기준 **월 약 750 ~ 850 USD(약 105 ~ 120만 원)** 수준으로 추정한다. 상시 가동 대비 컴퓨팅 비용이 약 60% 절감된 수치이며, 항목별 상세는 상세 기획안 11장을 따른다. 핵심 비용 통제 장치는 시간제 가동·자동 선셋·쿼터의 3중 구조다.

## 11. 기대 효과

- 심사 리드타임 단축: 수작업 수 일 → AI 심사 10분 + 보드 확인으로 1일 이내 목표
- 트리아지로 불필요한 환경 배정 자체를 차단 — 기존 도구로 가능한 과제는 도구 안내로 종결되어 자원·운영 부담이 신규 개발이 꼭 필요한 과제에 집중됨
- 환경 준비 수 주 → Terraform 자동 프로비저닝 30분 이내
- 시간제 가동·자동 선셋으로 유휴 비용 구조적 통제, 보드 운영 공수 최소화
- 계정 격리와 SCP 가드레일로 바텀업 과제의 장애·부하·보안 이슈가 탑다운(공식) 환경에 영향을 주지 않음
- 격상 파이프라인으로 검증된 현장 아이디어가 재작성 없이 전사 서비스로 확산
