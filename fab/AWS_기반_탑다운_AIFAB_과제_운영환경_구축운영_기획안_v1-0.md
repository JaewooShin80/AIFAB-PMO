# AWS 기반 탑다운(AIFAB) 과제 운영환경 구축·운영 기획안

AIFAB : 탑다운(Top-down) 공식 과제 운영 플랫폼

- 문서 구분 : 운영 기획(안)
- 버전 : v1.0
- 작성일 : 2026. 07. 11.
- 작성자 : 신재우
- 관련 문서 : AWS 기반 샌드박스 구축·운영 기획안 v2-1 (바텀업, 이하 "샌드박스 기획안")

---

## 1. 추진 배경 및 목적

### 1.1 추진 배경

- AIFAB 프로그램은 T2Y 달성을 목표로 하는 탑다운 AI 과제 수행을 위해 기획된 전사 공식 과제 체계임
- 탑다운 과제는 사업 계획으로 확정된 공식 과제로서 실사용자를 대상으로 상시 서비스를 제공해야 하므로, dev급 실험 환경(바텀업 샌드박스)과는 다른 운영급 환경이 필요함
- 바텀업 샌드박스와 동일한 Landing Zone(계정 거버넌스·인증·네트워크·보안·IaC)을 공유하되, 가동 방식·배포 체계·데이터 정책이 다른 별도 환경으로 운영함
- 바텀업 우수과제가 격상(마이그레이션)될 때 이를 수용하는 목적지 환경이기도 하므로, 격상 유입 절차를 갖춘 운영 체계가 필요함

### 1.2 목적

- 탑다운 공식 과제를 위한 상시 가동(24×365) 운영 환경을 계정 수준 격리로 제공
- 스테이징 → 운영 2단계 배포와 보드·정보보호팀 승인 게이트로 운영 반영의 품질·보안을 확보. 단, 게이트의 세부 운영 방식 — 스테이징 검증 단계의 정보보호팀 관여 수준, 재배포(변경 배포) 시 심사 범위·주기 — 은 정보보호팀과 협의하여 확정 필요
- 통제된 절차 하의 실데이터 연동을 지원하여 실제 업무 시스템 수준의 서비스 구현
- 바텀업 격상 과제의 이관 체크리스트·수용 절차 표준화
- 공통 Landing Zone 재사용으로 구축 기간·비용 최소화

### 1.3 적용 범위

과제 확정(사업 계획) → 환경 프로비저닝 → 개발 → 스테이징 검증 → 운영 배포 → 상시 운영·유지보수 → 종료 또는 이관에 이르는 탑다운 과제 전 생애주기와, 바텀업 격상 과제의 유입 절차를 대상으로 한다.

## 2. 바텀업 샌드박스와의 차이 (요약)

두 환경은 같은 플랫폼 기반 위에서 운영 등급만 달리한다. 상세 비교는 샌드박스 기획안 3.6을 따른다. **AI 사전심사 바이패스는 과제 적합성 심사에 한하며**, Bedrock 호출 로깅(Model Invocation Logging)·Guardrails·CloudTrail 감사 등 거버넌스 통제는 탑다운에도 동일하게 적용된다.

| **구분** | **탑다운 (본 기획)** | **바텀업 (샌드박스 기획안)** |
|:---|:---|:---|
| 과제 성격 | 사업 계획으로 확정된 공식 과제 | 현장 자율 제안 과제 |
| 심사 | AI 사전심사 바이패스(과제 적합성 심사에 한함), 보드 확인 | AI 다각도 심사·트리아지 후 보드 승인 |
| 실행 환경 | 별도 운영 환경(계정), 스테이징·운영 분리 | 개발과 동일 환경에 dev 배포 |
| 가동 | 상시 가동 (서비스 수준 관리) | 시간제 가동 (셧다운/웨이크업) |
| 배포 | 스테이징 → 운영 + 승인 게이트(보드·정보보호팀) | dev 배포까지 (게이트 없음) |
| 데이터 | 통제 하 실데이터 연동 허용 (민감정보 원본 포함) | 실데이터 허용 — 민감정보 마스킹 원칙 |
| 수행 주체 | 지정된 개발·운영 조직 | 시티즌 개발자 (교육 이수자) |
| 종료 | 사업 판단에 따른 종료·이관 | 저활용 자동 선셋 |

## 3. 계정·환경 구조

### 3.1 조직(OU)과 계정

샌드박스 기획안 3.2의 조직 구조를 공유하며, 본 기획의 대상은 Top-down OU다.

```
Root (관리 계정)
├── OU: Shared Services   — 공통 자동화·조직 로그 집적 (공용)
├── OU: Top-down          — 탑다운 과제 계정 (본 기획 대상)
└── OU: Bottom-up Sandbox — 바텀업 과제 계정 (샌드박스 기획안)
```

![그림 1. AWS 전체 아키텍처 (계정 분리 및 Top-Down/Bottom-up 이원 흐름)](media/fig1-aws-architecture.png)

*그림 1. AWS 전체 아키텍처 (계정 분리 및 Top-Down/Bottom-up 이원 흐름, 샌드박스 기획안과 공용)*

- 과제별 계정 분리를 기본으로 하고, 계정 안에서 스테이징·운영 환경을 분리한다(VPC 또는 서브넷 단위). 규모가 큰 과제는 스테이징·운영 계정 분리도 허용
- SCP는 공통 가드레일(리전 제한, 감사·보안 서비스 보호, S3 퍼블릭 노출 금지, 조직 이탈 금지 — 샌드박스 기획안 3.2.2의 1·4·5·6)을 동일 적용한다. 바텀업 전용 제약과 달리 시간제 가동은 적용하지 않으며, 네트워크 변경 제한(가드레일 3)은 운영 조직의 IaC 역할까지 예외로 확장
- **Top-down OU 추가 SCP 3종**: 탑다운 과제 계정은 실데이터와 운영 서비스를 다루므로, 아래 추가 Deny 가드레일을 Top-down OU에 부착한다.

| **#** | **가드레일** | **차단 대상** | **목적** |
|:---|:---|:---|:---|
| T-1 | KMS 키 즉시·단기 삭제 금지 | `kms:ScheduleKeyDeletion` — 삭제 대기 기간 30일 미만인 요청 거부 (`kms:ScheduleKeyDeletionPendingWindowInDays < 30` 조건) | 실데이터 암호화 키의 성급한 삭제로 인한 데이터 영구 손실 방지 |
| T-2 | Macie 비활성화 금지 | `macie2:DisableMacie` | 실데이터 취급 환경의 개인정보·민감정보 상시 탐지 체계 보호 |
| T-3 | CloudTrail 중지·삭제 금지 (강화) | `cloudtrail:StopLogging`, `cloudtrail:DeleteTrail` | 운영 환경 감사 기록의 무결성 보호 (공통 가드레일 4와 중복 적용으로 이중 차단) |

- 위 추가 SCP의 예외(IaC 자동화 경로)는 `aws:PrincipalArn` 조건으로 Shared Services의 Terraform 프로비저닝 역할 ARN에만 한정하며, 운영 조직 역할이나 개발자 권한세트에는 예외를 부여하지 않는다.

### 3.2 인증·권한

샌드박스 기획안 3.3(Identity Center · Entra ID 연동)을 그대로 공유하되, 권한세트를 운영 환경에 맞게 분리한다.

| **권한세트** | **대상** | **범위** |
|:---|:---|:---|
| TopdownDeveloper | 과제 개발자 | 스테이징 환경 개발 권한 (운영 환경 변경 불가) |
| TopdownOperator | 지정 운영 조직 | 운영 환경 모니터링·장애 대응 (인프라 변경은 IaC 경유) |
| PlatformAdmin | AI 인프라팀 | Landing Zone·공통 자동화 운영 (공용) |

### 3.3 실행 환경

- 과제 앱: ECS/Fargate 상시 태스크, **Multi-AZ 이중화** 기본. 오토스케일링으로 부하 대응 (최소 태스크 수 2개 기본, 과제별 SLO 확정 시 조정)
- 개발·스테이징: 개발 서버(EC2)와 스테이징 스택은 야간 자동 중지 등 비용 최적화 적용 가능 (운영 환경만 상시)
- IaC: 샌드박스 기획안 3.5의 Terraform 체계를 공유하되, 탑다운 전용 표준 모듈(`topdown-project`)로 Multi-AZ·모니터링·알람이 포함된 운영급 스택을 정의

#### 3.3.1 가용성·복구 기본안 (RPO/RTO)

탑다운 과제의 운영 수준 목표(SLO) 기본값은 다음과 같다. 과제별 SLO 확정 시 아래 기본안을 출발점으로 보드가 조정한다.

| **항목** | **기본안** | **구현 방법** |
|:---|:---|:---|
| RPO (목표 복구 시점) | 1시간 | RDS 자동 백업(보존 7일) + PITR(Point-in-Time Recovery) |
| RTO (목표 복구 시간) | 15분 | RDS Multi-AZ 자동 Failover, ECS 최소 태스크 2개 유지로 단일 AZ 장애 시 즉시 전환 |
| 최소 태스크 수 | 2개 (AZ 분산) | ECS 서비스 최소 수량 설정, Auto Scaling 최솟값 동일 적용 |
| 백업 보존 | 7일 | RDS 자동 백업 및 스냅샷, S3 버전 관리 활성화 |

## 4. 네트워크·보안

- 네트워크는 샌드박스 기획안 4장과 동일 원칙: 사내망 한정 접근(인터넷 인바운드 차단), Direct Connect/VPN + Transit Gateway 경유, VPC Endpoint로 AWS 서비스 통신, 아웃바운드 화이트리스트
- 탑다운 VPC는 과제 계정별 전용 대역을 할당하고 스테이징·운영 서브넷을 분리한다. 바텀업 VPC와는 계정·네트워크 모두 격리되어 상호 영향이 없음
- 보안 4계층(계정·접근, 네트워크, 데이터, 탐지·감사)과 Security Hub 통합·SNS 알림은 샌드박스 기획안 5장과 공통. CloudTrail·GuardDuty·Config·Inspector는 조직 전체 공통 계층으로 이미 적용됨

### 4.1 실데이터 취급 (바텀업과의 핵심 차이)

탑다운 과제는 통제된 절차 하에 민감정보 원본을 포함한 실데이터 연동을 허용한다 (바텀업은 실데이터를 허용하되 민감정보는 마스킹 원칙 — 민감정보 원본 연동은 탑다운에서만).

| **단계** | **통제** |
|:---|:---|
| 반입 승인 | 데이터 등급 분류 → 정보보호팀 사전 심사·승인 (연동 범위·보존 기간 명시) |
| 연동 방식 | 원본 복사 최소화 원칙 — 가능하면 API·조회 연동, 복제 시 KMS 암호화 저장 |
| 접근 통제 | 과제 계정 격리 + IAM 최소권한 + 접근 이력 전수 기록(CloudTrail) |
| 상시 탐지 | Macie로 등급 외 데이터·개인정보 유입 탐지, 위반 시 보드·정보보호팀 알림 |
| 종료 시 | 과제 종료·이관 시 데이터 파기 확인을 종료 체크리스트에 포함 |

#### 4.1.1 데이터 등급 분류 기준

데이터 반입 승인 절차의 전제로, 사내 정보보호 정책의 데이터 분류 등급 체계를 준용한다. **등급 체계 및 등급별 세부 기준은 정보보호팀 정책을 따르며**, 아래는 탑다운 과제 환경 적용을 위한 뼈대 구조다.

| **등급(예시)** | **정의(예시)** | **탑다운 반입 가능 여부** | **통제 수준** |
|:---|:---|:---|:---|
| 공개(Public) | 외부 공개 가능 정보 | 허용 | 기본 암호화 |
| 내부(Internal) | 임직원 공유 정보 | 허용 | KMS CMK 암호화, 접근 이력 기록 |
| 기밀(Confidential) | 사업상 민감 정보 | 정보보호팀 사전 심사 후 허용 | CMK + 접근 최소화 + Macie 감시 + 보존 기간 명시 |
| 극비(Restricted) | 법적·규제적 제약 정보 | 보드 + 정보보호팀 승인 필수 | 별도 격리 계정, 감사 강화 |

- 실제 등급 명칭·기준·절차는 정보보호팀 정책 문서를 원본으로 하며, 위 표는 과제 계획 단계 판단을 위한 참조 뼈대다.
- 반입 신청 시 데이터 등급을 명시하고, 기밀 이상 데이터는 정보보호팀이 연동 범위·보존 기간·파기 방법을 승인 조건으로 지정한다.

## 5. 개발·배포 (CI/CD)

과제별 Git 리포지토리에서 시작하는 파이프라인 구조는 샌드박스 기획안 7장과 동일하되, 탑다운은 운영 배포 단계와 승인 게이트가 존재한다.

![그림 2. 탑다운 CI/CD 및 배포 구조 (스테이징 → 승인 게이트 → 운영)](media/fig8-topdown-cicd.png)

*그림 2. 탑다운 CI/CD 및 배포 구조 (스테이징 → 승인 게이트 → 운영)*

| **단계** | **자동/수동** | **통과 기준** |
|:---|:---|:---|
| 빌드·테스트 | 자동 | 단위 테스트 통과, 하드코딩 시크릿 미검출 |
| 이미지 스캔 (ECR·Inspector) | 자동 | Critical/High 취약점 0건 (기준은 정보보호팀 협의 확정) |
| 스테이징 배포·검증 | 자동 + 오너 | 기능·성능 검증, 정책 준수 체크 통과 |
| **운영 배포 승인** | **수동** | **AI Board·정보보호팀 승인 게이트 통과** |
| 운영 배포 | 자동 | 무중단 배포(Blue/Green 또는 Rolling), 실패 시 자동 롤백 |
| 배포 후 모니터링 | 자동 | 오류율·응답시간 임계치 관리, 이상 시 알람·롤백 판단 |

- 운영 환경 직접 수정 경로는 존재하지 않는다(Fargate 불변 배포). 모든 변경은 Git 머지 → 파이프라인 재통과로만 반영
- 인프라 변경은 Terraform 코드 리뷰(PR) 기반으로만 수행
- **승인 게이트 운영 방식은 정보보호팀과 협의하여 확정한다 (협의 필요 사항)**: ① 최초 운영 배포 시 심사 범위·소요 기간 ② 재배포(변경 배포) 시 승인 절차 — 매 건 수동 심사 유지 또는 위험도 기반 차등(경미한 변경은 자동 검사 통과로 갈음, 보안 관련 변경만 수동 심사) ③ 스테이징 검증 단계의 정보보호팀 관여 수준

### 5.1 배포 방식 및 롤백 기본안

탑다운 운영 배포의 기본 방식은 **ECS Blue/Green(CODE_DEPLOY 컨트롤러)**이다. 새 태스크 셋(Green)이 준비되면 트래픽을 단계적으로 전환하며, 이상 감지 시 즉시 이전 버전(Blue)으로 복귀한다.

- **롤백 트리거 기본값**: 배포 완료 후 5분 이내 5xx 오류율 > 1% 또는 P99 응답시간 > 3초 (보드·정보보호팀 확정). CodeDeploy Deployment Group의 자동 롤백 설정에 위 조건을 CloudWatch Alarm으로 등록한다.
- **자동 롤백 훅**: CodeDeploy `AfterAllowTraffic` 라이프사이클 훅에서 상태 확인 Lambda를 실행하여, 임계치 초과 즉시 `StopDeployment` API를 호출하고 이전 태스크 셋으로 트래픽을 복귀시킨다.
- **수동 롤백**: 7장 런북의 "수동 롤백" 절차에 따라 운영 조직이 직전 정상 이미지 다이제스트를 지정하여 재배포한다.

### 5.2 아티팩트 관리 (ECR)

컨테이너 이미지는 Shared Services 계정의 중앙 ECR을 단일 원천으로 사용하고, 과제 계정은 크로스어카운트 Pull로 이미지를 가져온다.

- **크로스어카운트 Pull**: ECR 리포지토리 정책에 Top-down OU 계정의 `ecr:GetDownloadUrlForLayer` · `ecr:BatchGetImage`를 허용한다. 과제 계정의 ECS Task Execution Role에 해당 권한을 부여한다.
- **Lifecycle Policy**: 이미지 태그 정책 — `untagged` 이미지는 14일 후 자동 삭제, 태그된 이미지는 최근 10버전만 유지. `topdown-project` Terraform 모듈에 Lifecycle Policy를 기본 포함한다.
- **Tag Immutability + 다이제스트 고정 배포**: ECR 리포지토리에 Tag Immutability를 활성화하여 태그 덮어쓰기를 방지한다. ECS Task Definition의 이미지 참조는 `image-digest` 방식(SHA256)으로 지정하여, 같은 태그의 이미지가 교체되어도 배포 대상이 변경되지 않도록 한다.

## 7. 운영·모니터링

- 운영 수준 목표(SLO)를 과제별로 정의: 가용성(예: 99.5%), 오류율, 응답시간 — 목표치는 과제 오픈 시 보드가 확정
- CloudWatch 대시보드·알람을 표준 모듈에 포함하여 프로비저닝 시 자동 구성. 임계치 초과 시 운영 조직·AI 인프라팀에 SNS 알림
- 장애 대응: 운영 조직 1차 대응(Runbook 기반) → AI 인프라팀 인프라 지원 → 보안 사고 시 정보보호팀 공조
- 주간 리포트: 샌드박스 기획안 6.2의 리포트 에이전트를 공용으로 활용하여 탑다운 과제의 사용량·가용성·배포 이력을 보드·경영진에 보고
- 과제 종료·축소는 사업 판단(보드 심의)으로 결정하며, 종료 시 데이터 파기·자원 회수 체크리스트를 적용(자동 선셋은 적용하지 않음)

### 7.2 SSM Patch Manager (패치 관리)

탑다운 과제 계정의 EC2 인스턴스(개발 서버·스테이징 서버)에 대해 Systems Manager Patch Manager 기반 자동 패치를 적용한다.

- **패치 베이스라인**: AWS 관리형 기본 베이스라인을 사용하며, 사내 정보보호 정책상 요구 패치 승인 기준이 있을 경우 커스텀 베이스라인으로 정의(정보보호팀 협의 확정).
- **Maintenance Window**: 주 1회 정기 패치 윈도우 — 기본값 매주 일요일 01:00~03:00(서울 시간). 운영 환경 EC2가 있는 경우 별도 윈도우를 설정하여 서비스 중단 위험을 최소화한다.
- **알림**: 패치 완료 후 결과를 SNS로 AI 인프라팀 및 운영 조직에 통보. 실패 인스턴스는 수동 조치 후 재검증한다.

## 8. 단계별 구축 로드맵

공통 Landing Zone은 샌드박스 기획안 로드맵 1단계와 공동으로 구축하며, 본 기획의 추가 작업은 다음과 같다.

| **단계** | **기간(예시)** | **주요 작업** | **산출물** |
|:---|:---|:---|:---|
| 1단계: 공통 기반 (공동) | M1 ~ M2 | Landing Zone: Organizations·SCP, Identity Center, 네트워크, 공통 보안 (샌드박스 기획안과 동일 작업) | 격리된 계정 기반 환경 |
| 2단계: 탑다운 표준화 | M2 ~ M3 | `topdown-project` Terraform 모듈(Multi-AZ·모니터링 포함), 권한세트 분리, 실데이터 반입 절차 확정(정보보호팀 협의) | 탑다운 과제 환경 표준 |
| 3단계: 배포 체계 | M3 ~ M4 | CI/CD 표준 파이프라인, 승인 게이트, 무중단 배포·자동 롤백 | 운영급 배포 체계 |
| 4단계: 격상 수용·정착 | M4 ~ M6 | 격상 체크리스트·이관 자동화, SLO·Runbook 표준, 리포트 연계 | 격상 파이프라인 가동 |

- 스테이징·개발 자원의 업무 시간 외 자동 중지 (운영 환경은 제외)

- 과제 종료 시 Terraform destroy로 즉시 회수, S3 수명주기 정책 적용

### 9.4 비용 거버넌스 (태그 정책 + 예산 알람)

탑다운 과제의 비용을 과제별로 추적하고 상한을 강제하는 거버넌스 체계를 갖춘다.

- **표준 태그 강제**: Organizations Tag Policy로 아래 태그의 키 형식을 표준화하고, AWS Config `required-tags` 룰로 미태깅 자원을 탐지하여 SNS 알람을 발생시킨다. Terraform `topdown-project` 모듈이 자원 생성 시 표준 태그를 자동 부착한다.

| **태그 키** | **설명** | **예시 값** |
|:---|:---|:---|
| `project_id` | 과제 식별자 | `td-proj-001` |
| `owner` | 과제 오너 사번 | `EMP12345` |
| `env` | 환경 구분 | `staging`, `prod` |
| `cost_center` | 비용 귀속 부서 코드 | `CC-1234` |

- **계정별 Budgets 알람**: 과제 계정마다 월 예산을 설정하고, 예산의 80% 도달 시 경고 알람, 100% 도달 시 AI 인프라팀·과제 오너에게 SNS 알림을 전송한다. 예산 초과가 반복되는 과제는 보드 보고를 통해 자원 규모 조정 또는 쿼터 상향을 검토한다.

## 10. 기대 효과

- 공통 Landing Zone 재사용으로 탑다운 환경의 추가 구축 비용·기간 최소화 (증분 작업은 표준 모듈·배포 체계에 한정)
- 계정 수준 격리로 과제 간·바텀업과의 상호 영향 차단, 실데이터 취급 과제의 보안 경계 명확화
- 스테이징 → 운영 승인 게이트로 운영 반영 품질 보장, 모든 변경의 Git·파이프라인 이력 확보
- 바텀업 격상 과제를 재작성 없이 수용하는 파이프라인으로 현장 혁신의 전사 확산 경로 완성
- SLO·Runbook 기반 운영 표준화로 과제 수 증가에도 운영 부담의 선형 증가 방지
